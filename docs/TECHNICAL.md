# AI Memory MCP — Technical Documentation

## Stack (Docker Compose)

| Component | Source | Notes |
|-----------|--------|-------|
| Python | `mcp/Dockerfile` — `python:3.12-slim` | MCP service |
| FastMCP | `mcp/requirements.txt` | MCP over SSE (`mcp/main.py`) |
| PostgreSQL | `pgvector/pgvector:pg16` | DB `ai_memory`, extension `vector`, `pg_trgm` |
| Neo4j | `neo4j:5-community` | APOC; only on internal Docker network |
| Gitea | `gitea/gitea:1.23.4` | HTTP inside stack `:3000`; SSH **2222** published on host |
| Caddy | `caddy:2-alpine` | TLS, `caddy/Caddyfile` + `caddy/entrypoint.sh` |
| Admin UI | `admin/` (FastAPI) | Token/user management |

There is **no** host install of PostgreSQL, Gitea, or Caddy — everything runs in containers. Versions of Python libraries are pinned only in `mcp/requirements.txt` (no separate lockfile in repo).

## Network

- **`backend`**: `internal: true` — Postgres and Neo4j only here; not reachable from the host. MCP/Admin/Gitea also attach to **`proxy`** for Caddy and outbound API calls.
- **`proxy`**: Caddy, Gitea, MCP, Admin — fronted by Caddy on host `80`/`443`.

Databases are **not** mapped to host ports (`docker-compose.yml`). Access from the host to Postgres/Neo4j is only via `docker compose exec`, not `localhost:5432`.

## Services (containers)

| Container | Role | Host ports |
|-----------|------|------------|
| `caddy` | Reverse proxy, HTTPS | `80`, `443`, `443/udp` |
| `gitea` | Git HTTP + built-in SSH | `2222` → SSH (see `GITEA__server__SSH_PORT`) |
| `postgres` | Databases `ai_memory`, `gitea` | *(none)* |
| `neo4j` | Graph | *(none)* |
| `ai-memory-mcp` | MCP SSE | *(none; via Caddy)* |
| `ai-memory-admin` | Admin UI | *(none; via Caddy)* |

## Repository layout (deploy root)

```
./
  docker-compose.yml    # stack definition
  setup.sh              # guided .env + first deploy
  .env.example          # variable template
  caddy/
    Caddyfile
    entrypoint.sh       # whitelist → CADDY_ALLOWED_IPS
  mcp/                    # MCP image context
    main.py
    requirements.txt
    config.py
    tools/               # knowledge, graph, semantic, gitea_tools, git_tools
  admin/                  # Admin UI image context
  init-db/                # Postgres init (schema + first user)
```

Runtime config: **`.env`** in project root (generated by `setup.sh` or copied from `.env.example`). Not: `/etc/gitea`, `/etc/caddy` on the host — those paths apply only **inside** images if at all; this project mounts `./caddy/*` into the Caddy container.

## Database

- **DB name:** `ai_memory`
- **App DB user:** `mcpuser` (password `MCP_DB_PASSWORD` in `.env`)
- **Superuser** for dumps: `postgres` / `POSTGRES_PASSWORD`

### `categories` (excerpt)

`user_id` is required (multi-tenant). `path` is unique globally in schema (see `init-db/init.sh` for full DDL).

### Managing the stack

```bash
cd /path/to/ai-memory-mcp
docker compose ps
docker compose logs -f mcp
docker compose exec postgres psql -U mcpuser -d ai_memory
docker compose up -d --build mcp
docker compose down
```

## Caddy

- Config file in repo: `caddy/Caddyfile` — hostnames `gitea.{$SERVER_IP}.nip.io`, etc.
- Optional IP allowlist: `CADDY_WHITELIST_ENABLED`, `CADDY_WHITELIST_IPS` in `.env` (processed by `caddy/entrypoint.sh`).

## Category tree

`init-db/init.sh` creates schema and the first admin user only. The **example** tree in [README.md](README.md) is a recommended layout — create categories with `create_category` / Admin as needed.
